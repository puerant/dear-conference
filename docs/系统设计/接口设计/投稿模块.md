# 投稿模块分析

## 1. 模块概述

投稿模块负责管理学术会议的论文投稿流程，包括投稿提交、投稿列表查询、投稿详情查看、投稿修改、投稿删除等功能。审稿子模块负责管理审稿流程，包括审稿任务分配、审稿意见提交、审稿结果统计等。

该模块涉及两个主要角色：
- **投稿讲者（speaker）**：提交论文、查看审稿结果
- **审稿人（reviewer）**：审阅论文、提交审稿意见

## 2. 功能模块划分

### 2.1 投稿管理（讲者）
- 创建投稿
- 查看投稿列表
- 查看投稿详情
- 修改投稿（仅限待审稿状态）
- 删除投稿（仅限待审稿状态）
- 上传投稿附件

### 2.2 审稿管理（审稿人）
- 查看待审稿列表
- 查看已审稿列表
- 查看投稿详情
- 提交审稿意见
- 修改审稿意见

### 2.3 投稿管理（管理员）
- 查看所有投稿
- 分配审稿人
- 修改投稿状态
- 查看投稿统计

### 2.4 通知功能
- 投稿创建通知（邮件）
- 审稿分配通知（邮件）
- 审稿结果通知（邮件）

## 4. API实现逻辑

### 4.1 创建投稿逻辑

```
1. 接收参数：title, abstract, category, fileUrl(可选)
2. 权限校验：
   - 提取Token，获取user_id
   - 查询用户角色，必须为speaker
3. 参数校验：
   - title非空，长度1-200
   - abstract长度0-5000
   - category长度0-50
4. 创建投稿记录：
   - INSERT INTO submissions (user_id, title, abstract, category, file_url, status)
   - status默认为1（待审稿）
5. 发送邮件通知管理员有新投稿
6. 返回投稿ID
```

### 4.2 查询投稿列表逻辑

```
1. 接收参数：page, pageSize, status(可选), category(可选)
2. 权限校验：
   - 讲者：查询自己的投稿（WHERE user_id = 当前用户）
   - 管理员：查询所有投稿
3. 构建查询条件：
   - WHERE 1=1
   - AND user_id = ? (讲者)
   - AND status = ? (status非空时)
   - AND category = ? (category非空时)
4. 分页查询
5. 关联查询：
   - 关联users表获取讲者信息
   - 关联dictionary表获取状态名称
6. 返回分页结果
```

### 4.3 修改投稿逻辑

```
1. 接收参数：submissionId, title, abstract, category, fileUrl(可选)
2. 权限校验：
   - 提取Token，获取user_id
3. 查询投稿记录：
   - SELECT * FROM submissions WHERE id = ?
4. 验证：
   - 投稿必须存在
   - 必须是自己的投稿（user_id匹配）
   - 状态必须为1（待审稿），审稿中的投稿不可修改
5. 构建更新字段（只更新提供的参数）
6. 更新投稿：
   - UPDATE submissions SET ... WHERE id = ?
7. 返回更新成功
```

### 4.4 删除投稿逻辑

```
1. 接收参数：submissionId
2. 权限校验：
   - 提取Token，获取user_id
3. 查询投稿记录
4. 验证：
   - 投稿必须存在
   - 必须是自己的投稿
   - 状态必须为1（待审稿）
5. 删除投稿：
   - DELETE FROM submissions WHERE id = ?
6. 删除关联的审稿记录：
   - DELETE FROM reviews WHERE submission_id = ?
7. 返回删除成功
```

### 4.5 查看待审稿列表逻辑

```
1. 权限校验：
   - 提取Token，获取user_id
   - 查询用户角色，必须为reviewer
2. 查询待审稿投稿：
   - SELECT s.* FROM submissions s
     LEFT JOIN reviews r ON s.id = r.submission_id AND r.reviewer_id = ?
     WHERE s.status = 2 (reviewing)
       AND r.id IS NULL (未被该审稿人审过)
3. 关联users表获取讲者信息
4. 分页查询
5. 返回结果
```

### 4.6 提交审稿意见逻辑

```
1. 接收参数：submissionId, comment, result, score
2. 权限校验：
   - 提取Token，获取user_id
   - 用户角色必须为reviewer
3. 参数校验：
   - comment非空
   - result必须为1(accept),2(reject),3(revise)之一
   - score范围0-100
4. 检查是否已审稿：
   - SELECT count FROM reviews WHERE submission_id = ? AND reviewer_id = ?
5. 已审稿则不允许重复审稿
6. 插入审稿记录：
   - INSERT INTO reviews (submission_id, reviewer_id, comment, result, score)
7. 更新投稿状态（可选，根据业务规则）：
   - 可能需要等待多个审稿人都审完后才更新
8. 发送邮件通知讲者审稿结果
9. 返回审稿成功
```

### 4.7 分配审稿人逻辑

```
1. 接收参数：submissionId, reviewerIds (数组)
2. 权限校验：
   - 必须为admin角色
3. 查询投稿记录，验证存在
4. 检查审稿人是否为reviewer角色
5. 更新投稿状态为2（reviewing）
6. 为每个审稿人创建审稿任务记录（如需任务表）
7. 发送邮件通知审稿人
8. 返回分配成功
```

### 4.8 修改投稿状态逻辑

```
1. 接收参数：submissionId, status
2. 权限校验：必须为admin角色
3. 查询投稿记录
4. 检查所有审稿记录是否已提交
5. 更新投稿状态：
   - UPDATE submissions SET status = ? WHERE id = ?
6. 根据最终状态发送邮件通知讲者
7. 返回更新成功
```

## 5. 模块关联关系

### 5.1 关联的模块

| 关联模块 | 关联方式 | 关联说明 |
|---------|---------|---------|
| 用户模块 | user_id, reviewer_id | 投稿关联讲者，审稿关联审稿人 |
| 字典模块 | status, result | 投稿状态、审稿结果使用字典表维护 |

### 5.2 数据流向

```
投稿流程（讲者）：
登录 → 创建投稿接口 → submissions表 → 邮件通知管理员 → 分配审稿人

审稿流程（审稿人）：
登录 → 查看待审稿列表 → 查看投稿详情 → 提交审稿意见 → reviews表

结果通知流程：
审稿提交 → 发送邮件 → 讲者查看投稿详情 → 看到审稿结果
```

## 6. 业务规则

### 6.1 投稿规则
1. 只有speaker角色可以创建投稿
2. 投稿标题必填，长度1-200字符
3. 摘要选填，最多5000字符
4. 投稿分类选填，如"人工智能"、"大数据"等
5. 附件大小限制10MB
6. 只有status=1（待审稿）的投稿可以修改或删除

### 6.2 审稿规则
1. 只有reviewer角色可以审稿
2. 一个审稿人对同一投稿只能审稿一次
3. 审稿意见必填
4. 审稿结果：accept（录用）、reject（拒绝）、revise（需修改）
5. 评分范围0-100分
6. 投稿状态为reviewing时才可审稿

### 6.3 投稿状态流转
```
pending (待审稿)
    ↓ 分配审稿人
reviewing (审稿中)
    ↓ 所有审稿人完成
accepted (已录用) / rejected (已拒绝)
```

### 6.4 审稿分配规则
1. 管理员手动分配审稿人
2. 一个投稿可以分配多个审稿人
3. 审稿人角色验证（必须为reviewer）
4. 分配后投稿状态自动变为reviewing

## 7. 异常处理

| 异常码 | 异常信息 | 说明 |
|--------|---------|------|
| 2001 | 投稿不存在 | 查询的投稿不存在 |
| 2002 | 无权操作此投稿 | 不是投稿的创建者 |
| 2003 | 投稿状态不允许修改 | 只有待审稿状态可修改 |
| 2004 | 投稿状态不允许删除 | 只有待审稿状态可删除 |
| 2005 | 已提交审稿，不可重复审稿 | 审稿人已审过该投稿 |
| 2006 | 审稿人数量不足 | 分配审稿人时参数错误 |
| 2007 | 审稿人角色无效 | 指定的用户不是审稿人 |
| 2008 | 投稿标题过长 | 超过200字符限制 |
| 2009 | 附件大小超限 | 超过10MB限制 |

## 9. 数据库操作

### 9.1 主要查询语句

```sql
-- 查询我的投稿列表（讲者）
SELECT s.*, u.name as speaker_name
FROM submissions s
LEFT JOIN users u ON s.user_id = u.id
WHERE s.user_id = ?
ORDER BY s.created_at DESC
LIMIT ?, ?;

-- 查看待审稿列表（审稿人）
SELECT s.*, u.name as speaker_name
FROM submissions s
LEFT JOIN reviews r ON s.id = r.submission_id AND r.reviewer_id = ?
LEFT JOIN users u ON s.user_id = u.id
WHERE s.status = 2
  AND r.id IS NULL
ORDER BY s.created_at ASC
LIMIT ?, ?;

-- 查看投稿详情
SELECT s.*, u.name as speaker_name, u.email as speaker_email
FROM submissions s
LEFT JOIN users u ON s.user_id = u.id
WHERE s.id = ?;

-- 查询投稿的审稿记录
SELECT r.*, u.name as reviewer_name
FROM reviews r
LEFT JOIN users u ON r.reviewer_id = u.id
WHERE r.submission_id = ?
ORDER BY r.created_at DESC;

-- 创建投稿
INSERT INTO submissions (user_id, title, abstract, category, file_url, status)
VALUES (?, ?, ?, ?, 1);

-- 提交审稿
INSERT INTO reviews (submission_id, reviewer_id, comment, result, score)
VALUES (?, ?, ?, ?, ?);

-- 更新投稿状态
UPDATE submissions SET status = ?, updated_at = NOW() WHERE id = ?;
```

### 9.2 外键维护（应用层）

```java
// 检查投稿是否存在
public boolean submissionExists(Long submissionId) {
    return submissionMapper.selectCountById(submissionId) > 0;
}

// 检查是否是投稿创建者
public boolean isOwner(Long submissionId, Long userId) {
    Submission submission = submissionMapper.selectById(submissionId);
    return submission != null && submission.getUserId().equals(userId);
}

// 检查用户是否为审稿人
public boolean isReviewer(Long userId) {
    User user = userService.getById(userId);
    return user != null && user.getRole() == 2; // 2 = reviewer
}

// 检查是否已审稿
public boolean hasReviewed(Long submissionId, Long reviewerId) {
    return reviewMapper.selectCountBySubmissionAndReviewer(submissionId, reviewerId) > 0;
}
```

## 10. 文件上传处理

### 10.1 文件上传逻辑

```
1. 接收文件参数：file
2. 验证文件：
   - 文件类型：pdf/doc/docx
   - 文件大小：<= 10MB
3. 生成文件名：{UUID}.{扩展名}
4. 上传到OSS：
   - 路径：/submissions/{year}/{month}/{文件名}
5. 获取文件URL
6. 返回：{ fileUrl }
```

### 10.2 文件删除逻辑

```
1. 接收文件URL
2. 解析文件路径
3. 删除OSS上的文件
4. 返回删除成功
```
