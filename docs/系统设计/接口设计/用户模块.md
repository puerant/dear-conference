# 用户模块分析

## 1. 模块概述

用户模块是系统的核心模块之一，负责管理用户的注册、登录、个人信息维护、角色权限控制等功能。系统支持四种角色：投稿讲者（speaker）、审稿人（reviewer）、参会者（attendee）、管理员（admin）。

## 2. 功能模块划分

### 2.1 注册登录
- 用户注册（邮箱注册）
- 用户登录（邮箱+密码）
- 登出
- 密码重置
- 邮箱验证

### 2.2 个人信息管理
- 查看个人信息
- 修改个人信息
- 上传/修改头像
- 修改密码

### 2.3 角色权限
- 角色分配（管理员操作）
- 权限校验
- 角色切换（单用户多角色场景预留）

### 2.4 用户管理（管理员）
- 用户列表查询
- 用户状态管理（启用/禁用）
- 用户角色修改
- 用户删除

## 4. API实现逻辑

### 4.1 用户注册逻辑

```
1. 接收参数：email, password, name, role(可选)
2. 参数校验：
   - 邮箱格式校验
   - 密码强度校验（至少8位，包含字母和数字）
   - 姓名非空校验
3. 检查邮箱是否已存在
   - SELECT count FROM users WHERE email = ?
4. 邮箱未注册：
   - 密码BCrypt加密
   - 插入用户记录
   - 生成邮箱验证码
   - 发送验证邮件
5. 返回注册成功，提示验证邮箱
```

### 4.2 用户登录逻辑

```
1. 接收参数：email, password
2. 参数校验：
   - 邮箱和密码非空
3. 根据邮箱查询用户：
   - SELECT * FROM users WHERE email = ? AND status = 1
4. 用户不存在或状态禁用：
   - 返回"用户不存在或已被禁用"
5. 验证密码：
   - BCrypt.checkpw(plainPassword, hashedPassword)
6. 密码验证通过：
   - 生成JWT Token（有效期24小时）
   - 封装用户信息（排除密码）
   - 返回 { token, userInfo }
```

### 4.3 修改个人信息逻辑

```
1. 接收参数：name, avatar(可选)
2. 权限校验：Token提取user_id
3. 构建更新对象（只允许修改name和avatar）
4. 更新用户信息：
   - UPDATE users SET name=?, avatar=?, updated_at=NOW() WHERE id=?
5. 返回更新成功
```

### 4.4 修改密码逻辑

```
1. 接收参数：oldPassword, newPassword
2. 权限校验：Token提取user_id
3. 查询用户当前密码
4. 验证旧密码：
   - BCrypt.checkpw(oldPassword, hashedPassword)
5. 新密码强度校验
6. 新密码BCrypt加密
7. 更新密码：
   - UPDATE users SET password=? WHERE id=?
8. 返回更新成功
```

### 4.5 用户列表查询逻辑

```
1. 接收参数：page, pageSize, role(可选), status(可选), keyword(可选)
2. 构建查询条件：
   - WHERE 1=1
   - AND role = ? (role非空时)
   - AND status = ? (status非空时)
   - AND (name LIKE ? OR email LIKE ?) (keyword非空时)
3. 查询总数：SELECT count(*) FROM users WHERE ...
4. 分页查询：SELECT * FROM users WHERE ... LIMIT ?, ?
5. 关联字典表查询角色名称
6. 关联字典表查询状态名称
7. 返回分页结果
```

### 4.6 修改用户状态逻辑

```
1. 权限校验：必须为admin角色
2. 接收参数：userId, status(0或1)
3. 检查用户是否存在：
   - SELECT count FROM users WHERE id = ?
4. 检查是否修改自己：
   - 禁止管理员禁用自己
5. 更新状态：
   - UPDATE users SET status=? WHERE id=?
6. 返回更新成功
```

## 5. 模块关联关系

### 5.1 关联的模块

| 关联模块 | 关联方式 | 关联说明 |
|---------|---------|---------|
| 投稿模块 | user_id | 投稿表关联讲者用户 |
| 审稿模块 | user_id | 审稿记录表关联审稿人用户 |
| 订单模块 | user_id | 订单表关联下单用户 |
| 字典模块 | role, status | 用户角色和状态使用字典表维护 |

### 5.2 数据流向

```
注册流程：
用户 → 注册接口 → users表 → 发送验证邮件 → 验证邮箱 → 激活账户

登录流程：
用户 → 登录接口 → 验证用户信息 → 生成JWT Token → 返回Token

投稿流程（讲者）：
登录 → 获取Token → 调用投稿接口 → 投稿表记录user_id

审稿流程（审稿人）：
登录 → 获取Token → 调用审稿接口 → 审稿记录表记录reviewer_id

下单流程（参会者）：
登录 → 获取Token → 调用下单接口 → 订单表记录user_id
```

## 6. 业务规则

### 6.1 注册规则
1. 邮箱作为唯一登录账号
2. 密码必须使用BCrypt加密存储
3. 注册后需验证邮箱才能登录
4. 默认角色为attendee（参会者）
5. 管理员创建用户时可以指定角色

### 6.2 登录规则
1. 只有状态为1（激活）的用户可以登录
2. 密码错误次数限制（建议5次）
3. JWT Token有效期24小时
4. 登录成功后Token存储到localStorage

### 6.3 权限规则
1. speaker（投稿讲者）：可以创建投稿，查看自己的投稿
2. reviewer（审稿人）：可以查看待审稿列表，提交审稿意见
3. attendee（参会者）：可以购买票务，查看订单和凭证
4. admin（管理员）：拥有所有权限

### 6.4 信息修改规则
1. 用户只能修改自己的信息
2. 邮箱不可修改（如需修改需重新注册或联系管理员）
3. 头像URL存储到OSS或其他文件服务
4. 密码修改需要验证旧密码

## 7. 异常处理

| 异常码 | 异常信息 | 说明 |
|--------|---------|------|
| 1001 | 邮箱已被注册 | 注册时邮箱已存在 |
| 1002 | 密码格式不正确 | 密码不符合强度要求 |
| 1003 | 用户不存在或密码错误 | 登录失败 |
| 1004 | 用户已被禁用 | 用户账号被禁用 |
| 1005 | 旧密码错误 | 修改密码时旧密码验证失败 |
| 1006 | 无权限操作 | 权限不足 |
| 1007 | 不能禁用自己 | 管理员尝试禁用自己 |

## 9. 数据库操作

### 9.1 主要查询语句

```sql
-- 根据邮箱查询用户
SELECT * FROM users WHERE email = ? AND status = 1;

-- 根据ID查询用户
SELECT * FROM users WHERE id = ?;

-- 分页查询用户列表
SELECT * FROM users
WHERE 1=1
  AND role = ?
  AND status = ?
  AND (name LIKE ? OR email LIKE ?)
ORDER BY created_at DESC
LIMIT ?, ?;

-- 更新用户信息
UPDATE users
SET name = ?, avatar = ?, updated_at = NOW()
WHERE id = ?;

-- 修改密码
UPDATE users
SET password = ?, updated_at = NOW()
WHERE id = ?;

-- 修改用户状态
UPDATE users
SET status = ?, updated_at = NOW()
WHERE id = ?;

-- 删除用户
DELETE FROM users WHERE id = ?;
```
