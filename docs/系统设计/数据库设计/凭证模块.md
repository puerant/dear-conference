# 凭证模块数据库设计

## 1. 模块概述

凭证模块负责管理参会者的入会凭证，包括二维码生成、座位分配、凭证核销（使用）等功能。凭证是订单支付成功后自动生成的入会证明。

## 2. 表结构

### 2.1 入会凭证表 (credentials)

```sql
CREATE TABLE `credentials` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `order_id` BIGINT NOT NULL COMMENT '订单ID',
  `credential_no` VARCHAR(32) NOT NULL COMMENT '凭证号',
  `qr_code` VARCHAR(500) COMMENT '二维码数据',
  `seat_info` VARCHAR(50) COMMENT '座位信息',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态:1-valid,2-used,3-expired',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `used_at` DATETIME COMMENT '使用时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_credential_no` (`credential_no`),
  KEY `idx_order_id` (`order_id`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='入会凭证表';
```

## 3. 字段说明

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| id | BIGINT | 是 | 主键ID，自增 |
| order_id | BIGINT | 是 | 关联的订单ID，关联orders表 |
| credential_no | VARCHAR(32) | 是 | 凭证号，唯一标识，格式：CRD+时间戳+随机数 |
| qr_code | VARCHAR(500) | 否 | 二维码数据，用于现场扫码核销 |
| seat_info | VARCHAR(50) | 否 | 座位信息，如"A区3排5座" |
| status | TINYINT | 是 | 凭证状态，关联字典表 dict_code=credential_status |
| created_at | DATETIME | 是 | 凭证生成时间（订单支付成功时） |
| used_at | DATETIME | 否 | 凭证核销（使用）时间 |

## 4. 索引说明

| 索引名 | 类型 | 字段 | 说明 |
|--------|------|------|------|
| PRIMARY | 主键 | id | 主键索引 |
| uk_credential_no | 唯一 | credential_no | 凭证号唯一性约束 |
| idx_order_id | 普通 | order_id | 查询订单的所有凭证 |
| idx_status | 普通 | status | 查询有效的凭证 |

## 5. 数据字典

### 5.1 凭证状态 (dict_code: credential_status)

| item_code | item_value | item_name | 说明 |
|-----------|------------|-----------|------|
| valid | 1 | 有效 | 凭证正常，可以参加会议 |
| used | 2 | 已使用 | 凭证已核销，无法重复使用 |
| expired | 3 | 已过期 | 订单退款后凭证失效 |

## 6. 业务约束

### 6.1 凭证生成约束
1. **订单关联**：order_id必须关联orders表中status=paid的订单
2. **生成数量**：凭证数量 = 订单的quantity数量
3. **唯一性**：credential_no在系统内必须唯一

### 6.2 凭证核销约束
1. **状态限制**：只有status=1（有效）的凭证可以核销
2. **一次性使用**：凭证核销后不可重复使用
3. **核销记录**：核销时记录used_at时间

### 6.3 凭证失效约束
1. **订单退款**：订单退款后，关联凭证状态更新为expired
2. **会议结束**：会议结束后可将有效凭证标记为used或保留为valid（根据业务规则）

## 7. 业务流程

### 7.1 凭证生成流程
```
订单支付成功
    ↓
根据订单quantity生成对应数量的凭证
    ↓
为每个凭证生成唯一的credential_no
    ↓
生成二维码数据 qr_code
    ↓
分配座位信息 seat_info (如果有)
    ↓
状态设置为 valid
    ↓
发送凭证到用户邮箱
```

### 7.2 凭证核销流程
```
参会者出示二维码
    ↓
工作人员扫码
    ↓
验证凭证状态 (必须为valid)
    ↓
验证凭证时效
    ↓
核销凭证
    ↓
更新状态 status: used
    ↓
记录核销时间 used_at
```

### 7.3 凭证失效流程
```
订单退款成功
    ↓
查询订单关联的所有凭证
    ↓
批量更新凭证状态 status: expired
    ↓
失效凭证无法使用
```

## 8. 凭证号生成规则

格式：`CRD{timestamp}{random}`

示例：`CRD20260225143000654321`

- CRD：凭证前缀
- 20260225143000：时间戳 (年月日时分秒)
- 654321：6位随机数

## 9. 二维码格式建议

二维码数据建议格式（JSON）：

```json
{
  "credential_no": "CRD20260225143000654321",
  "attendee_name": "张三",
  "ticket_name": "普通票",
  "seat_info": "A区3排5座",
  "status": "valid",
  "expire_time": "2026-03-31 18:00:00"
}
```

## 10. 座位分配规则

### 10.1 座位信息格式
- 按区域编号：A/B/C/D区
- 按排号：1-20排
- 按座号：1-50座

示例：`A区3排5座`、`VIP区1排1座`

### 10.2 分配策略
1. **普通票**：按A、B、C区域顺序分配
2. **VIP票**：分配VIP区域前排座位
3. **早鸟票**：优先分配前排座位

## 11. 关联模块

- **订单模块**：凭证关联订单，订单支付后生成凭证
- **用户模块**：凭证信息中的参会人信息关联订单中的参会人信息
- **字典模块**：使用字典表维护凭证状态
