## 零、票种类型说明

本系统支持两种票种类型，在创建票务时由管理员指定，购票行为和流程因类型不同而有所差异。

### 0.1 个人票（ticket_type = 1）

| 特性 | 说明 |
|------|------|
| 购买数量 | 每次固定购买 1 张，不可更改 |
| 参会者信息 | 自动从登录用户信息中读取（姓名、邮箱），无需重复填写 |
| 下单流程 | 一步完成：确认票务 → 生成订单 → 支付 → 生成凭证 |
| 凭证生成 | 支付成功后立即生成 1 张入会凭证 |

### 0.2 团体票（ticket_type = 2）

| 特性 | 说明 |
|------|------|
| 购买数量 | 可购买多张（受 min_persons / max_persons 限制） |
| 参会者信息 | 分两阶段录入，第一阶段仅填写团体负责人信息，成员信息异步补录 |
| 下单流程 | 两阶段（见 0.2.1） |
| 凭证生成 | 每个成员槽填写完毕后生成对应凭证，未填写的槽暂不生成 |

#### 0.2.1 团体票购票两阶段流程

```
第一阶段（负责人操作，同步）
─────────────────────────────────────────────────────────────
① 负责人选择团体票 → 填写团体名称、联系人、购买人数
② 系统锁定对应库存（orders 创建，status=UNPAID）
   同时在 group_members 表创建 N 条空白成员槽
③ 负责人完成支付
④ 支付成功后系统生成 invite_token，可对外分享邀请链接

第二阶段（成员信息补录，异步）
─────────────────────────────────────────────────────────────
方式 A：负责人在「我的团体」管理页面手动录入/修改/删除成员
  - 负责人自行填写每位成员的姓名和邮箱
  - 可随时增加（未超过购买人数上限）或删除成员信息

方式 B：分享邀请链接，由成员本人填写
  - 系统生成专属邀请链接（含 invite_token），负责人分享给成员
  - 成员无需登录，通过链接填写自己的姓名和邮箱，占用一个成员槽
  - 成员提交后不可自行修改，如需修改须联系负责人（方式 A）操作

成员信息填写完毕后，系统自动为该成员生成入会凭证（QR码）
```

## 二、核心业务流程

### 2.1 查询票务列表

```
前端 Portal /tickets 或 Admin /ticket
  → GET /api/ticket?page=1&pageSize=10&status=1
  → TicketController.list()
      - 接收 page、pageSize、status（可选）参数
      - LambdaQueryWrapper 构建条件
      - status 非空时加入条件：status = ?
      - 分页查询：ORDER BY sort_order ASC, created_at DESC
  → 计算可用库存：available = stock - sold_count
  → 返回 PageResult<TicketVo>（含 ticketType、minPersons、maxPersons 字段）
```

### 2.2 查询票务详情

```
前端 Portal /tickets/:id
  → GET /api/ticket/{id}
  → TicketController.getById(id)
      - 根据 id 查询票务
      - 票务不存在抛 TICKET_NOT_FOUND(4001)
  → 计算可用库存：available = stock - sold_count
  → 返回 TicketVo（含 available、ticketType、minPersons、maxPersons 字段）
```

### 2.3 查询可购买票务

```
前端 Portal /tickets
  → GET /api/ticket/available?page=1&pageSize=10
  → TicketController.listAvailable()
      - 条件：status = 1 AND stock > sold_count
      - 分页查询
  → 返回 PageResult<TicketVo>
```

### 2.4 管理员 - 创建票务

```
前端 Admin /ticket/create
  → POST /api/admin/ticket
  → AdminTicketController.create()
      - @RequireRole("admin") 权限校验
      - CreateTicketDto 参数校验：
          - name：1-100字符，非空
          - price：>0，小数点后2位
          - stock：>= 0
          - ticketType：1（个人票）或 2（团体票），默认 1
          - minPersons：团体票时必填，>= 2；个人票忽略（固定为 1）
          - maxPersons：团体票时可选，>= minPersons；为 null 表示不限上限
          - description：0-1000字符
      - 插入 tickets 表：sold_count = 0, status = 1
  → 返回创建的票务信息
```

### 2.5 管理员 - 修改票务信息

```
前端 Admin /ticket/:id/edit
  → PUT /api/admin/ticket/{id} { name?, price?, description?, sortOrder?, minPersons?, maxPersons? }
  → AdminTicketController.update()
      - @RequireRole("admin") 权限校验
      - 查询票务是否存在，不存在抛 TICKET_NOT_FOUND(4001)
      - ticketType 不允许修改（已有订单时）
      - 更新提供的字段
  → 返回更新后的票务信息
```

### 2.6 管理员 - 修改票务状态

```
前端 Admin /ticket 列表操作栏「上架/下架」按钮
  → PUT /api/admin/ticket/{id}/status { status: 0|1 }
  → AdminTicketController.updateStatus()
      - @RequireRole("admin") 权限校验
      - 查询票务
      - 下架前检查：status=1 → 0 时，确保无未完成订单（可选）
      - 更新状态
  → 返回成功
```

### 2.7 管理员 - 调整库存

```
前端 Admin /ticket 列表「调整库存」按钮
  → PUT /api/admin/ticket/{id}/stock { stock: 数量 }
  → AdminTicketController.updateStock()
      - @RequireRole("admin") 权限校验
      - 验证：新库存 >= sold_count
      - 验证失败抛 STOCK_ADJUST_INVALID(4007)
      - 更新 stock
  → 返回成功
```

### 2.8 管理员 - 删除票务

```
前端 Admin /ticket 列表「删除」按钮
  → DELETE /api/admin/ticket/{id}
  → AdminTicketController.delete()
      - @RequireRole("admin") 权限校验
      - 查询票务
      - 检查：sold_count == 0，否则抛 TICKET_SOLD_CANT_DELETE(4006)
      - 删除票务
  → 返回成功
```

### 2.9 管理员 - 票务统计

```
前端 Admin /dashboard 或 /statistics
  → GET /api/admin/ticket/statistics
  → AdminTicketController.getStatistics()
      - @RequireRole("admin") 权限校验
      - 查询：
          - totalCount：总票务数
          - activeCount：上架票务数
          - totalStock：总库存
          - totalSold：已售数量
          - totalSales：总销售额（sum(price * sold_count)）
  → 返回统计数据
```

### 2.10 个人票 - 下单流程

```
前端 Portal /tickets/:id → 点击「立即购票」
  → POST /api/orders
     {
       "ticketId": 1,
       "orderType": 1         // 个人票，quantity 固定为 1 无需传递
     }
  → OrderController.create()
      - @RequireRole("attendee") 权限校验
      - 查询票务，校验 ticket_type == 1（个人票）
      - 数量固定为 1；attendee_name / attendee_email 从当前登录用户信息自动获取
      - 库存原子扣减 1（decreaseStock）
      - 创建 orders 记录：order_type=1，status=UNPAID
  → 返回订单信息，跳转到支付页面
  → 支付成功后 CredentialService 为该订单生成 1 张入会凭证
```

### 2.11 团体票 - 第一阶段：创建团体订单

```
前端 Portal /tickets/:id → 点击「团体购票」→ 填写团体信息
  → POST /api/orders
     {
       "ticketId": 2,
       "orderType": 2,
       "quantity": 10,           // 购买人数（受 minPersons/maxPersons 约束）
       "groupName": "清华大学计算机系",
       "contactName": "张三",
       "contactEmail": "zhangsan@thu.edu.cn"
     }
  → OrderController.create()
      - @RequireRole("attendee") 权限校验
      - 查询票务，校验 ticket_type == 2（团体票）
      - 验证 quantity ∈ [minPersons, maxPersons]（maxPersons 为 null 则不校验上限）
          - 不符合抛 GROUP_QUANTITY_INVALID(3010)
      - 库存原子扣减 quantity（decreaseStock）
      - 创建 orders 记录：
          order_type=2, status=UNPAID,
          group_name, contact_name（从 contactName 取），
          attendee_email（从 contactEmail 取），
          invite_token 暂不生成（支付后生成）
      - 在 group_members 表批量插入 quantity 条空白成员槽（status=1，待填写）
  → 返回订单信息，跳转到支付页面

  支付成功后：
      - 生成 UUID 作为 invite_token，更新到 orders 表
      - 不立即生成凭证（等待成员信息填写）
```

### 2.12 团体票 - 成员信息管理（负责人操作）

#### 获取团体订单详情

```
前端 Portal /orders/:id/group（「我的订单」→「管理成员」）
  → GET /api/orders/{id}/group
  → GroupOrderController.getGroupDetail()
      - @RequireLogin，校验订单属于当前用户
      - 查询 orders + group_members 列表
  → 返回 GroupOrderVo：
     {
       "orderId": 1,
       "groupName": "清华大学计算机系",
       "contactName": "张三",
       "totalCount": 10,
       "filledCount": 3,
       "inviteUrl": "https://domain.com/group/invite/{token}",
       "members": [
         { "id": 1, "memberName": "李四", "memberEmail": "lisi@...", "status": 2 },
         { "id": 2, "memberName": null, "memberEmail": null, "status": 1 },
         ...
       ]
     }
```

#### 获取/生成邀请链接

```
前端 Portal /orders/:id/group → 点击「分享邀请链接」
  → GET /api/orders/{id}/group/invite
  → GroupOrderController.getInviteLink()
      - @RequireLogin，校验订单属于当前用户，且 status=PAID
      - 若 invite_token 已存在，直接返回
      - 若 invite_token 为空，生成 UUID 后更新到 orders 表再返回
  → 返回 { "inviteUrl": "https://domain.com/group/invite/{token}" }
```

#### 负责人添加成员

```
前端 Portal /orders/:id/group → 点击「添加成员」
  → POST /api/orders/{id}/group/members { "memberName": "王五", "memberEmail": "wangwu@..." }
  → GroupOrderController.addMember()
      - @RequireLogin，校验订单属于当前用户，且 status=PAID
      - 查询当前已填写成员数 filledCount
      - filledCount >= quantity 时抛 GROUP_MEMBER_FULL(3011)
      - 找到第一个 status=1（待填写）的成员槽，更新信息并设 status=2
      - 触发凭证生成：CredentialService.createForMember(orderId, memberId)
  → 返回更新后的 GroupMemberVo
```

#### 负责人修改成员

```
前端 Portal 成员列表 → 点击「编辑」
  → PUT /api/orders/{id}/group/members/{memberId}
     { "memberName": "王五修改后", "memberEmail": "new@..." }
  → GroupOrderController.updateMember()
      - @RequireLogin，校验订单属于当前用户
      - 校验 memberId 属于该订单
      - 更新姓名/邮箱，filled_at 更新为当前时间
      - 若邮箱有变更，同步更新关联凭证的收件地址（或标记凭证需重发邮件）
  → 返回更新后的成员信息
```

#### 负责人删除成员信息

```
前端 Portal 成员列表 → 点击「清除」
  → DELETE /api/orders/{id}/group/members/{memberId}
  → GroupOrderController.clearMember()
      - @RequireLogin，校验订单属于当前用户
      - 将该成员槽重置为空白：memberName=null，memberEmail=null，status=1，filled_at=null
      - 将关联凭证（如已生成）状态置为 REVOKED
  → 返回成功
```

### 2.13 团体票 - 成员自助填写（邀请链接）

#### 访问邀请页面获取团体信息

```
前端 Portal /group/invite/:token（成员通过分享链接访问，无需登录）
  → GET /api/group/invite/{token}
  → GroupInviteController.getGroupInfo()
      - 无需 JWT 鉴权
      - 通过 invite_token 查询对应 orders 记录
      - token 不存在抛 INVITE_TOKEN_INVALID(3012)
      - 订单 status != PAID 抛 GROUP_NOT_PAID(3013)
      - 计算剩余可填写槽数：remainSlots = quantity - filledCount
  → 返回：
     {
       "groupName": "清华大学计算机系",
       "contactName": "张三",
       "ticketName": "团体票",
       "totalCount": 10,
       "filledCount": 7,
       "remainSlots": 3      // 0 时前端显示「名额已满」
     }
```

#### 成员自助提交信息

```
前端 Portal 邀请页 → 成员填写姓名和邮箱后提交
  → POST /api/group/invite/{token}/fill { "memberName": "赵六", "memberEmail": "zhaoliu@..." }
  → GroupInviteController.fillMemberInfo()
      - 无需 JWT 鉴权
      - 查询 orders（通过 token），校验状态=PAID
      - 检查剩余槽数 > 0，否则抛 GROUP_MEMBER_FULL(3011)
      - 找第一个 status=1 的成员槽（SELECT ... FOR UPDATE 防并发抢占同一槽）
      - 更新该槽：memberName, memberEmail, status=2, filled_at=now()
      - 触发凭证生成：CredentialService.createForMember(orderId, memberId)
  → 返回：{ "message": "信息已提交，入会凭证将通过邮件发送至您的邮箱" }

  注意：
  - 同一邮箱可重复提交（不做唯一校验，允许多人共邮箱场景）
  - 提交成功后，成员无法通过此链接再次修改；如需修改联系负责人
```

## 三、数据库变更（团体票功能）

### 3.1 tickets 表新增字段

```sql
ALTER TABLE tickets
  ADD COLUMN ticket_type  TINYINT NOT NULL DEFAULT 1
      COMMENT '票种类型：1=个人票 2=团体票' AFTER description,
  ADD COLUMN min_persons  INT     DEFAULT NULL
      COMMENT '团体票最小购买人数（个人票为 NULL）' AFTER ticket_type,
  ADD COLUMN max_persons  INT     DEFAULT NULL
      COMMENT '团体票最大购买人数（NULL 表示不限）' AFTER min_persons;
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `ticket_type` | TINYINT | 1 | 1=个人票，2=团体票 |
| `min_persons` | INT | NULL | 团体票最小购买人数，个人票为 NULL |
| `max_persons` | INT | NULL | 团体票最大购买人数，NULL 表示不限 |

### 3.2 orders 表新增字段

```sql
ALTER TABLE orders
  ADD COLUMN order_type    TINYINT     NOT NULL DEFAULT 1
      COMMENT '订单类型：1=个人订单 2=团体订单' AFTER status,
  ADD COLUMN group_name    VARCHAR(100) DEFAULT NULL
      COMMENT '团体名称（团体订单必填）' AFTER order_type,
  ADD COLUMN invite_token  VARCHAR(64)  DEFAULT NULL
      COMMENT '团体邀请链接 Token（支付后生成）' AFTER group_name,
  ADD UNIQUE KEY uk_invite_token (invite_token);
```

> **字段复用说明**：
> - 个人订单：`attendee_name` / `attendee_email` 存参会者本人信息（从用户 profile 自动填充）
> - 团体订单：`attendee_name` / `attendee_email` 存团体联系人信息（由负责人填写）；`group_name` 存团体名称

| 字段 | 类型 | 说明 |
|------|------|------|
| `order_type` | TINYINT | 1=个人订单，2=团体订单 |
| `group_name` | VARCHAR(100) | 团体名称，团体订单必填 |
| `invite_token` | VARCHAR(64) UNIQUE | 邀请链接 Token，支付成功后生成 |

### 3.3 新增 group_members 表

```sql
CREATE TABLE group_members (
    id           BIGINT      NOT NULL AUTO_INCREMENT COMMENT '主键',
    order_id     BIGINT      NOT NULL               COMMENT '关联订单 ID',
    sequence_no  INT         NOT NULL               COMMENT '成员序号（1~quantity，在订单内唯一）',
    member_name  VARCHAR(50) DEFAULT NULL           COMMENT '成员姓名',
    member_email VARCHAR(100) DEFAULT NULL          COMMENT '成员邮箱',
    status       TINYINT     NOT NULL DEFAULT 1     COMMENT '1=待填写 2=已填写',
    filled_at    DATETIME    DEFAULT NULL            COMMENT '填写时间',
    created_at   DATETIME    NOT NULL               COMMENT '创建时间（随订单一起批量创建）',
    PRIMARY KEY (id),
    UNIQUE KEY uk_order_seq (order_id, sequence_no),
    INDEX idx_order_id (order_id),
    INDEX idx_status   (order_id, status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='团体订单成员信息';
```

凭证生成逻辑：

| 场景 | 触发时机 | 凭证数量 |

## 八、关联错误码

| 错误码 | 错误信息 | 说明 |
|--------|---------|------|
| 4001 | 票务不存在 | 查询的票务不存在（实际错误码，文档原标注 3001 有误） |
| 3002 | 票务名称过长 | 超过100字符限制 |
| 3003 | 票务价格无效 | 价格必须大于0 |
| 3004 | 库存不足 | 购买数量超过可用库存 |
| 4006 | 票务已售出不能删除 | sold_count > 0（实际错误码，文档原标注 3005 有误） |
| 4007 | 库存调整无效 | 调整后库存低于已售数量（实际错误码，文档原标注 3006 有误） |
| 3007 | 票务有未完成订单不能下架 | 存在未支付或支付中的订单 |
| 3008 | 票种类型不匹配 | 使用团体票接口购买个人票或反之 |
| 3009 | 个人票不支持多张购买 | 个人票 quantity 必须为 1 |
| 3010 | 购买人数不符合团体票要求 | quantity 不在 [minPersons, maxPersons] 范围内 |
| 3011 | 团体成员已满 | 所有成员槽均已填写，无空余槽位 |
| 3012 | 邀请链接无效 | invite_token 不存在或已失效 |
| 3013 | 团体订单未完成支付 | 通过邀请链接访问时订单尚未支付 |
