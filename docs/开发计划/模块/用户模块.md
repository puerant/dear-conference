## 二、核心业务流程

### 2.1 用户注册（含邮箱验证）

```
前端 Portal /register
  → POST /api/auth/register { email, password, name, role }
  → RegisterDto 校验（邮箱格式、密码8位含字母数字、role 只允许 speaker/attendee）
  → AuthServiceImpl.register()
      - 检查邮箱是否已存在 → 已存在抛 USER_ALREADY_EXISTS(1002)
      - BCrypt 加密密码
      - INSERT INTO users（email_verified=0）
      - VerificationCodeService.generateAndStore(email)
          → 生成6位随机验证码，存入 Redis（key: verify:code:{email}，TTL 15min）
          → 同时设置频率限制（key: verify:limit:{email}，TTL 60s）
      - EmailService.sendVerificationCode(email, code)
          → dev-mode=true：打印日志
          → dev-mode=false：Hutool MailUtil 发送 HTML 邮件
  → 返回 { code: 200, message: "注册成功，验证码已发送到您的邮箱" }
  → 前端跳转至 /verify-email?email={email}

前端 Portal /verify-email
  → 用户输入6位验证码，点击「验证邮箱」
  → POST /api/auth/verify-email { email, code }
  → AuthServiceImpl.verifyEmail()
      - 查找用户是否存在
      - VerificationCodeService.verify(email, code)
          → 比对 Redis 存储的验证码，正确则删除 key
          → 不正确抛 VERIFY_CODE_INVALID(1007)
      - UPDATE users SET email_verified=1
      - 生成 JWT（有效期 24h）
  → 返回 { token, userInfo }
  → 前端存储 token/userInfo，跳转首页
```

**关键约束：**
- 注册后账号处于 `email_verified=0` 状态，**无法登录**（登录时校验邮箱验证状态）
- 验证码6位纯数字，Redis TTL 15分钟，发送限频60秒/次
- 邮件通过 Hutool `MailUtil` + 163 SMTP（SSL port 465）发送

---

### 2.2 重发验证码

```
前端 /verify-email 页面「重新发送」按钮（60秒倒计时）
  → POST /api/auth/send-code { email }
  → AuthServiceImpl.sendVerificationCode()
      - 检查用户是否存在
      - isRateLimited(email) → Redis verify:limit:{email} 若存在则抛 VERIFY_CODE_TOO_FREQUENT(1008)
      - 重新生成验证码并发送
```

---

### 2.3 用户登录

```
前端 Portal 或 Admin /login
  → POST /api/auth/login { email, password }
  → AuthServiceImpl.login()
      - 查询 users WHERE email = ?
      - BCrypt.checkpw(plainPassword, hashedPassword)
      - status == 0 → 抛 USER_DISABLED(1005)
      - emailVerified == 0 → 抛 EMAIL_NOT_VERIFIED(1006)（须先完成邮箱验证）
      - 生成 JWT（有效期 24h，包含 userId / role）
  → 返回 { token, userInfo: { id, email, name, role, avatar } }

前端存储：
  Admin  → localStorage.setItem('admin_token', token)
  Portal → localStorage.setItem('portal_token', token)
```

**路由守卫（已修复）：**
- Admin `beforeEach`：校验 `admin_token` 存在 **且** `role === 'admin'`，否则跳回 `/login`
- Portal `beforeEach`：校验 `portal_token` 存在，且对有 `meta.roles` 的路由验证角色

### 2.4 Token 刷新

```
客户端（Axios 响应拦截器）收到 code 401
  → 检查 localStorage 中是否有 token（_retried 标记防死循环）
  → 调用 POST /api/auth/refresh（携带旧 token，绕过 @RequireLogin）
  → AuthServiceImpl.refresh()
      - jwtUtil.getClaimsAllowExpired(token)
          → 正常 token：直接解析
          → 已过期 token：捕获 ExpiredJwtException 取 claims
          → 签名无效等其他异常：抛 TOKEN_INVALID(1004)
      - 检查宽限期：仅允许过期 24h 内的 token 刷新
      - 查库验证用户仍存在且未被禁用
      - 签发新 JWT（从刷新时刻起 24h）
  → 返回 { token, userInfo }
  → 前端更新 localStorage + Pinia store
  → 用新 token 重放原失败请求
  → 多个并发 401 请求：第一个触发刷新，其余排队等待新 token
```

### 2.4 个人信息修改 / 密码修改

```
PUT /api/user/profile
  { name?, avatar?, oldPassword?, newPassword? }
  → UpdateUserDto 接收参数
  → 若含 oldPassword + newPassword：
      - BCrypt 验证旧密码 → 不匹配抛 PASSWORD_ERROR(1003)
      - BCrypt 加密新密码并更新
  → 若仅含 name / avatar：直接更新对应字段
  → 返回更新后的 userInfo
```

---

### 2.5 管理员 - 用户列表与状态管理

```
GET /api/admin/users?page=1&pageSize=10&role=reviewer&keyword=张
  → AdminUserController.list()
  → UserRole.ofCode(role) 将字符串 role code 转为枚举（已修复）
  → LambdaQueryWrapper 构建条件：role = ? AND (name LIKE ? OR email LIKE ?)
  → 返回 PageResult<User>（password 字段置 null）

PUT /api/admin/users/{id}/status  { status: 0|1 }
  → 更新 users.status
  → status=0（禁用）后该账号登录时 AuthServiceImpl 查不到激活用户，返回业务错误
```

---

### 2.6 管理员 - 创建用户 / 修改角色 / 删除用户

```
POST /api/admin/users  { email, password, name, role }
  → AdminCreateUserDto 校验（邮箱格式、密码6-20位、role 允许所有角色）
  → 检查邮箱是否已存在 → 已存在抛 USER_ALREADY_EXISTS(1002)
  → BCrypt 加密密码
  → 直接设置 email_verified=1、status=1（管理员创建无需邮箱验证）
  → INSERT INTO users

PUT /api/admin/users/{id}/role  { role: "reviewer" }
  → 查找用户，若不存在抛 USER_NOT_FOUND(1001)
  → UserRole.ofCode(role) 转为枚举
  → 更新 users.role

DELETE /api/admin/users/{id}
  → 查找用户，若不存在抛 USER_NOT_FOUND(1001)
  → userMapper.deleteById(id)
```
