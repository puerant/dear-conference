## 一、模块目标

文件模块从系统模块中独立，统一负责平台全部文件资产的上传、存储、访问、归档与治理。

本次重构重点新增 **OSS 管理中心**，支持接入多家云存储供应商，并实现按业务分类路由到不同存储。

### 1.1 核心目标

1. 支持多供应商并存（本地 + 阿里云 OSS + 腾讯云 COS + MinIO/S3 兼容）。
2. 支持同一系统维护多个存储配置（如生产桶、备份桶、私有桶）。
3. 文件上传时按策略自动选择存储位置，支持手动指定供应商。
4. 文件元数据与物理存储解耦，支持后续文件迁移与容灾切换。
5. 文件访问统一通过签名 URL 或网关下载，避免暴露真实存储路径。

### 1.2 功能范围

| 功能域 | 子功能 | 说明 |
|------|------|------|
| 文件资产管理 | 上传/查询/删除/恢复/预览 | 管理文件全生命周期 |
| 文件校验能力 | 类型校验、大小校验、MD5 去重 | 防止恶意文件和重复存储 |
| OSS 供应商管理 | 新增、编辑、启停、测试连接 | 支持多供应商配置 |
| 存储桶管理 | Bucket 管理、目录前缀、访问域名 | 精细化管理对象存储空间 |
| 存储策略管理 | 默认策略、按分类路由、私有/公有 | 文件按业务落盘 |
| 文件迁移治理 | 跨供应商迁移、失败重试、任务记录 | 支持供应商切换与成本优化 |

## 二、设计原则

1. 供应商抽象：通过统一 `StorageClient` 接口屏蔽各厂商差异。
2. 配置解耦：供应商配置、Bucket、路由策略分别建模。
3. 安全优先：AccessKey/SecretKey 等敏感信息加密存储（AES）。
4. 向后兼容：保留现有 `system_file` 主表，新增关联字段与扩展表。
5. 可运维：所有关键操作记录审计日志和迁移任务日志。

## 三、数据库设计（新增/调整）

> 说明：以下为文件模块建议表结构，最终 SQL 建议落地到 `docs/开发文档/sql/migration/` 新脚本（如 `08_file_module_oss.sql`）。

### 3.1 供应商配置表 `storage_provider`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| provider_name | VARCHAR(100) | 配置名称（如“阿里云-主存储”） |
| provider_type | TINYINT | 1=LOCAL,2=ALI_OSS,3=TENCENT_COS,4=MINIO_S3 |
| endpoint | VARCHAR(255) | 访问端点 |
| region | VARCHAR(100) | 区域 |
| access_key | VARCHAR(255) | AK（加密存储） |
| secret_key | VARCHAR(255) | SK（加密存储） |
| sts_enabled | TINYINT | 是否启用 STS（预留） |
| is_enabled | TINYINT | 是否启用 |
| is_default | TINYINT | 是否默认供应商 |
| remark | VARCHAR(255) | 备注 |
| created_at / updated_at | DATETIME | 时间戳 |

约束建议：
- 同一时刻仅允许一个 `is_default=1 AND is_enabled=1`。
- 禁止删除被 Bucket/策略引用的供应商。

### 3.2 存储桶表 `storage_bucket`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| provider_id | BIGINT | 关联 `storage_provider.id` |
| bucket_name | VARCHAR(128) | Bucket 名称 |
| base_path | VARCHAR(255) | 基础目录前缀（如 `conference/`） |
| domain | VARCHAR(255) | 访问域名（CDN 或 OSS 域名） |
| acl_type | TINYINT | 1=private,2=public-read |
| is_enabled | TINYINT | 是否启用 |
| is_default | TINYINT | 该供应商下默认桶 |
| created_at / updated_at | DATETIME | 时间戳 |

### 3.3 存储策略表 `file_storage_policy`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| policy_name | VARCHAR(100) | 策略名称 |
| file_category | VARCHAR(64) | 文件分类（conference_banner 等） |
| file_type | VARCHAR(32) | image/document/presentation |
| provider_id | BIGINT | 指定供应商 |
| bucket_id | BIGINT | 指定 bucket |
| path_rule | VARCHAR(255) | 路径规则，如 `yyyy/MM/dd/{category}` |
| max_size_mb | INT | 单文件大小限制 |
| allowed_ext | VARCHAR(255) | 允许后缀，逗号分隔 |
| priority | INT | 优先级（越小越高） |
| is_enabled | TINYINT | 是否启用 |
| created_at / updated_at | DATETIME | 时间戳 |

### 3.4 文件主表扩展 `system_file`（在现有基础上新增）

新增字段建议：
- `provider_id`：实际存储供应商
- `bucket_id`：实际存储桶
- `object_key`：对象存储 key（不含域名）
- `storage_path`：完整存储路径（内部使用）
- `visibility`：1=private,2=public
- `version_no`：版本号（预留）
- `is_migrated`：是否迁移过

### 3.5 文件迁移任务表 `file_migration_task`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| task_no | VARCHAR(64) | 任务号 |
| source_provider_id | BIGINT | 源供应商 |
| target_provider_id | BIGINT | 目标供应商 |
| file_count | INT | 总文件数 |
| success_count | INT | 成功数 |
| fail_count | INT | 失败数 |
| status | TINYINT | 1=待执行,2=执行中,3=完成,4=失败 |
| error_message | VARCHAR(1000) | 失败原因 |
| created_by | BIGINT | 创建人 |
| started_at / finished_at | DATETIME | 执行时间 |
| created_at / updated_at | DATETIME | 时间戳 |

## 四、文件分类与路由策略

延续现有 `dictionary_item(dict_code=file_category)`，并建议补充：

| item_value | item_name | 说明 |
|-----------|-----------|------|
| 8 | paper_attachment | 投稿附件 |
| 9 | review_attachment | 审稿附件 |
| 10 | conference_material | 会议资料 |

路由优先级：
1. 上传请求显式指定 `providerId/bucketId`（管理员可用）。
2. 命中 `file_storage_policy`（按分类/类型匹配，priority 最小优先）。
3. 回退到默认供应商 + 默认 bucket。
4. 无可用目标时返回 `FILE_STORAGE_TARGET_NOT_FOUND`。

## 五、API 接口设计

统一前缀：`/api/admin/file`

### 5.1 文件管理

1. `POST /upload`：上传文件（multipart）
2. `GET /list`：分页查询文件
3. `DELETE /{id}`：软删除
4. `POST /{id}/restore`：恢复
5. `GET /{id}/preview-url`：生成预览 URL（私有对象签名）
6. `POST /batch-delete`：批量软删除

上传参数新增：
- `providerId`（可选）
- `bucketId`（可选）
- `visibility`（可选，private/public）

### 5.2 OSS 供应商管理

1. `GET /providers`：供应商列表
2. `POST /providers`：新增供应商
3. `PUT /providers/{id}`：编辑供应商
4. `PUT /providers/{id}/enable`：启停供应商
5. `PUT /providers/{id}/default`：设为默认
6. `POST /providers/{id}/test`：测试连通性（鉴权 + bucket 探测）
7. `DELETE /providers/{id}`：删除供应商

### 5.3 Bucket 管理

1. `GET /buckets?providerId=xx`
2. `POST /buckets`
3. `PUT /buckets/{id}`
4. `PUT /buckets/{id}/enable`
5. `PUT /buckets/{id}/default`
6. `DELETE /buckets/{id}`

### 5.4 策略管理

1. `GET /policies`
2. `POST /policies`
3. `PUT /policies/{id}`
4. `PUT /policies/{id}/enable`
5. `DELETE /policies/{id}`

### 5.5 文件迁移

1. `POST /migration/tasks`：创建迁移任务
2. `GET /migration/tasks`：任务列表
3. `GET /migration/tasks/{id}`：任务详情
4. `POST /migration/tasks/{id}/retry`：失败重试

### 6.2 关键流程

#### 上传流程
1. 校验文件大小、后缀、MIME。
2. 计算 MD5，命中去重则直接返回历史记录。
3. `FileRoutingService` 解析目标供应商与 bucket。
4. 通过 `StorageClientFactory` 获取对应客户端并上传。
5. 落库 `system_file`（记录 provider/bucket/object_key）。

#### 预览流程
1. 查询文件记录。
2. public 对象直接返回拼接 URL。
3. private 对象通过对应供应商 SDK 生成短时签名 URL（默认 10 分钟）。

#### 迁移流程
1. 扫描源供应商文件列表（分页）。
2. 读取源对象流 -> 写入目标供应商。
3. 成功后更新 `system_file` 的 provider/bucket/object_key。
4. 记录任务进度和失败明细。

## 八、安全与合规

1. 敏感字段（AK/SK、STS 凭证）必须 AES 加密存储。
2. 预览 URL 必须短期有效，默认 `600s`。
3. 上传后缀与 MIME 双重校验，拒绝可执行脚本上传。
4. 文件删除默认软删除；物理删除由定时任务延迟清理。
5. 记录操作日志：供应商新增/编辑/启停/删除、策略变更、迁移任务。

## 十二、兼容与迁移建议

1. 保留原 `system_file.storage_type` 字段，短期与 `provider_id` 并行。
2. 历史文件可通过脚本回填到“默认本地供应商”配置下。
3. 原接口 `/api/admin/system/file/*` 可保留一版兼容，内部转发到新模块。
4. 分阶段发布：先上线供应商管理与新上传链路，再启用迁移任务。

## 十三、与系统模块关系

- 系统模块保留“配置中心”职责，不再承载文件实现细节。
- 文件相关能力统一收敛到文件模块，系统模块中仅保留入口导航说明。
- 邮箱模块、凭证模块、投稿模块、票务模块统一调用文件模块接口，不直接访问底层存储 SDK。
