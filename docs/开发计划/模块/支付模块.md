## 一、设计目标与边界

### 1.1 目标

将当前「点击即成功」的**模拟支付**替换为真实支付网关对接，支持以下四种支付渠道：

| 渠道 | 适用人群 | 货币 | 接入方式 |
|------|---------|------|---------|
| 微信支付（WeChat Pay） | 国内用户 | CNY | Native Pay v3（PC 二维码） |
| 支付宝（Alipay） | 国内用户 | CNY | PC 网页支付（alipay.trade.page.pay） |
| 银联（UnionPay） | 国内用户 | CNY | 银联在线支付网关 |
| PayPal | 海外用户 | USD | Checkout Orders REST API v2 |

### 1.2 边界

- 本文档仅覆盖**支付发起**和**回调处理**流程，退款沿用现有 `POST /api/admin/orders/{id}/refund`，但需扩展为调用各渠道退款 API
- 不涉及账单对账、分账、T+N 结算等财务深度功能
- 汇率换算（CNY→USD）采用**系统配置固定汇率**，不接入实时汇率接口

### 2.2 设计模式：策略模式

```java
// 统一支付策略接口（实际实现）
public interface PaymentStrategy {
    String getProviderCode();          // "wechat" / "alipay" / "unionpay" / "paypal"
    boolean isEnabled();               // 是否已配置且启用；工厂仅注册 isEnabled()=true 的策略
    PaymentResponse createPayment(PayRequest req);
    NotifyResult parseNotify(HttpServletRequest request, String rawBody);
    void refund(RefundRequest req);
    String buildSuccessResponse();     // 各渠道回调成功应答格式不同
    String buildFailResponse();
}

// 工厂：仅注册 isEnabled()=true 的策略，避免未配置渠道被误调用
@Component
public class PaymentStrategyFactory {
    private final Map<String, PaymentStrategy> strategies;

    public PaymentStrategyFactory(List<PaymentStrategy> list) {
        this.strategies = list.stream()
                .filter(PaymentStrategy::isEnabled)
                .collect(Collectors.toMap(PaymentStrategy::getProviderCode, Function.identity()));
    }

    public PaymentStrategy get(String providerCode) {
        PaymentStrategy strategy = strategies.get(providerCode);
        if (strategy == null) throw new BusinessException(ErrorCode.PAYMENT_PROVIDER_NOT_FOUND);  // 7001
        return strategy;
    }
}
```

> **实现说明**：所有 4 个 Strategy 都注册为 Spring Bean，但 `UnionPayStrategy`/`PayPalStrategy` 的 `isEnabled()` 始终返回 `false`，因此不会出现在工厂 map 中。SDK 初始化代码在 `afterPropertiesSet()` 内通过 `isEnabled()` 保护，不会因未配置凭证而导致启动失败。

## 三、数据库设计

### 3.1 新增表：`payment_records`（支付记录）

```sql
CREATE TABLE payment_records (
    id               BIGINT       NOT NULL AUTO_INCREMENT         COMMENT '主键',
    payment_no       VARCHAR(64)  NOT NULL UNIQUE                 COMMENT '内部支付单号（PAY+时间戳+随机数）',
    order_id         BIGINT       NOT NULL                        COMMENT '关联订单 ID',
    order_no         VARCHAR(64)  NOT NULL                        COMMENT '订单号（冗余，方便对账）',
    provider         VARCHAR(20)  NOT NULL                        COMMENT '支付渠道：wechat/alipay/unionpay/paypal',
    provider_trade_no VARCHAR(128)                                COMMENT '第三方交易流水号（回调后填入）',
    amount           DECIMAL(10,2) NOT NULL                       COMMENT '支付金额（CNY）',
    currency         VARCHAR(10)  NOT NULL DEFAULT 'CNY'          COMMENT '实际支付货币',
    exchange_rate    DECIMAL(10,4)                                COMMENT 'CNY→目标货币汇率（PayPal 使用）',
    foreign_amount   DECIMAL(10,2)                                COMMENT '外币金额（PayPal，单位 USD）',
    status           TINYINT      NOT NULL DEFAULT 1              COMMENT '1=待支付 2=支付成功 3=支付失败 4=已关闭 5=已退款',
    pay_url          TEXT                                         COMMENT '支付跳转链接（Alipay/UnionPay/PayPal）',
    qr_code          TEXT                                         COMMENT '二维码内容（WeChat Native Pay）',
    expire_at        DATETIME                                     COMMENT '支付超时时间（默认15分钟）',
    notify_raw       MEDIUMTEXT                                   COMMENT '回调原始报文（留存备查）',
    created_at       DATETIME     NOT NULL                        COMMENT '创建时间',
    paid_at          DATETIME                                     COMMENT '支付成功时间',
    PRIMARY KEY (id),
    INDEX idx_order_id    (order_id),
    INDEX idx_payment_no  (payment_no),
    INDEX idx_provider_no (provider_trade_no)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='支付记录';
```

### 3.2 新增表：`refund_records`（退款记录）

```sql
CREATE TABLE refund_records (
    id                 BIGINT       NOT NULL AUTO_INCREMENT   COMMENT '主键',
    refund_no          VARCHAR(64)  NOT NULL UNIQUE           COMMENT '内部退款单号（REF+时间戳）',
    payment_no         VARCHAR(64)  NOT NULL                  COMMENT '关联支付单号',
    order_id           BIGINT       NOT NULL                  COMMENT '关联订单 ID',
    provider           VARCHAR(20)  NOT NULL                  COMMENT '支付渠道',
    provider_refund_no VARCHAR(128)                           COMMENT '第三方退款流水号',
    amount             DECIMAL(10,2) NOT NULL                 COMMENT '退款金额（CNY）',
    status             TINYINT      NOT NULL DEFAULT 1        COMMENT '1=处理中 2=退款成功 3=退款失败',
    reason             VARCHAR(255)                           COMMENT '退款原因',
    created_at         DATETIME     NOT NULL                  COMMENT '申请时间',
    refunded_at        DATETIME                               COMMENT '退款到账时间',
    PRIMARY KEY (id),
    INDEX idx_payment_no (payment_no),
    INDEX idx_order_id   (order_id)

```

### 3.3 新增枚举：`PaymentStatus`

| 值 | 常量 | 说明 |
|----|------|------|
| 1 | PENDING | 待支付（已发起，等待用户操作） |
| 2 | SUCCESS | 支付成功 |
| 3 | FAILED | 支付失败（用户主动关闭/超时） |
| 4 | CLOSED | 已关闭（系统主动关闭，如超时15分钟） |
| 5 | REFUNDED | 已退款 |

## 四、接口设计

### 4.1 发起支付

**接口变更说明**：原 `POST /api/orders/{id}/pay` 由「模拟立即成功」改为「发起支付，返回支付数据」。

```
POST /api/orders/{id}/pay
权限：@RequireRole("attendee")
Content-Type: application/json

请求体：
{
  "method": "wechat"   // wechat | alipay | unionpay | paypal
}

响应（成功）：
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "paymentNo": "PAY20260228153000123",
    "method": "wechat",
    "status": "pending",
    "qrCode": "weixin://wxpay/bizpayurl?pr=...",   // 微信专用，前端生成二维码图片
    "payUrl": null,                                  // Alipay/UnionPay/PayPal 使用
    "amount": 299.00,
    "currency": "CNY",
    "foreignAmount": null,                           // PayPal 专用（USD）
    "expireAt": "2026-02-28T15:45:00"
  }
}

// PayPal 响应示例：
{
  "data": {
    "paymentNo": "PAY20260228153000456",
    "method": "paypal",
    "payUrl": "https://www.paypal.com/checkoutnow?token=...",
    "amount": 299.00,
    "currency": "CNY",
    "foreignAmount": 41.53,     // 299 / 7.20（汇率）
    "expireAt": "2026-02-28T16:00:00"
  }
}
```

**幂等处理**：同一订单若存在 `status=PENDING` 的支付记录，直接返回已有记录（不重复发起）。

**错误码**：

| code | message | 触发条件 |
|------|---------|---------|
| 3001 | 订单不存在 | 订单 ID 不存在 |
| 3002 | 订单状态错误 | 订单非 UNPAID 状态 |
| 3005 | 无权访问该订单 | 非本人订单 |
| 7001 | 不支持的支付方式 | method 参数非法 |
| 7002 | 支付渠道暂不可用 | 调用渠道 API 失败 |

### 4.2 查询支付状态（前端轮询）

```
GET /api/payment/{paymentNo}/status
权限：@RequireLogin

响应：
{
  "code": 200,
  "data": {
    "paymentNo": "PAY20260228153000123",
    "status": "pending",    // pending | success | failed | closed
    "paidAt": null
  }
}
```

前端轮询策略：每 3 秒查询一次（实际实现为 3000ms），收到 `success` 后关闭弹窗并刷新订单列表；收到 `closed`/`failed` 后提示"支付已失效"。

---

### 4.3 回调接收（各渠道 Webhook，无 JWT 鉴权）

```
POST /api/payment/notify/wechat      (微信回调，XML/JSON 报文 + 微信签名)
POST /api/payment/notify/alipay      (支付宝回调，form-urlencoded + RSA 签名)
POST /api/payment/notify/unionpay    (银联回调，form-urlencoded + 证书签名)
POST /api/payment/notify/paypal      (PayPal Webhook，JSON + PAYPAL-TRANSMISSION-SIG)
```

> ⚠️ 这些端点**无需 `@RequireRole`/`@RequireLogin` 注解**即可被外部访问（AOP 层不拦截无注解的端点）。`JwtAuthenticationFilter` 仅在 Token 有效时设置请求属性，不会阻断无 Token 的请求，因此**无需修改白名单配置**。但必须严格**验签**，拒绝任何未通过签名校验的请求。

---

### 4.4 退款（Admin，扩展现有接口）

```
POST /api/admin/orders/{id}/refund
权限：@RequireRole("admin")

新增行为（在现有逻辑基础上追加）：
  ① 查找该订单对应的 payment_records（status=SUCCESS）
  ② 生成内部退款单号 REF{timestamp}
  ③ 调用对应渠道的退款 API
  ④ INSERT INTO refund_records
  ⑤ 更新 payment_records.status=REFUNDED
  ⑥ 后续逻辑不变（订单 REFUNDED、凭证 EXPIRED、库存回滚）
```

> **注意**：各渠道退款到账时间不同（微信 1-3 个工作日，支付宝实时~5 天，PayPal 3-5 个工作日），退款单状态需异步回调更新，或定时轮询渠道退款查询接口。

## 五、各渠道接入细节

### 5.1 微信支付（WeChat Pay Native v3）

#### 依赖
```xml
<dependency>
    <groupId>com.github.wechatpay-apiv3</groupId>
    <artifactId>wechatpay-java</artifactId>
    <version>0.2.14</version>
</dependency>
```

#### 配置项
```yaml
payment:
  wechat:
    enabled: true
    app-id: wx1234567890abcdef          # 公众号/小程序 AppID
    mch-id: "1234567890"               # 商户号
    api-v3-key: your-api-v3-key-32chars # API v3 密钥（32字节）
    private-key-path: classpath:cert/apiclient_key.pem  # 商户私钥
    cert-serial-no: your-cert-serial   # 商户证书序列号
    notify-url: https://yourdomain.com/api/payment/notify/wechat
```

#### 支付流程
```
1. 调用微信 API：POST https://api.mch.weixin.qq.com/v3/pay/transactions/native
   请求体：{
     "appid": "${appId}",
     "mchid": "${mchId}",
     "description": "会议报名-{ticketName}",
     "out_trade_no": "{paymentNo}",
     "notify_url": "{notifyUrl}",
     "amount": { "total": {分为单位的金额}, "currency": "CNY" }
   }
2. 响应：{ "code_url": "weixin://wxpay/bizpayurl?pr=xxx" }
3. 将 code_url 存入 payment_records.qr_code，返回前端
4. 前端用 QRCode.js 生成二维码图片展示

回调验签（使用 WechatPayHttpClientBuilder 的自动验签）：
  - 从 HTTP Header 提取：Wechatpay-Timestamp、Wechatpay-Nonce、Wechatpay-Signature、Wechatpay-Serial
  - SDK 自动完成 RSA-SHA256 验签
  - 解密 AES-256-GCM 加密的通知报文
  - 从报文中提取 out_trade_no（内部支付单号）和 transaction_id（微信流水号）
  - 更新 payment_records.provider_trade_no=transaction_id，status=SUCCESS
  - 返回：HTTP 200 + { "code": "SUCCESS", "message": "成功" }
```

#### 退款
```
调用：POST https://api.mch.weixin.qq.com/v3/refund/domestic/refunds
请求体：{
  "out_trade_no": "{paymentNo}",
  "out_refund_no": "{refundNo}",
  "amount": { "refund": {退款金额（分）}, "total": {原支付金额（分）}, "currency": "CNY" }
}
```

### 5.2 支付宝（Alipay PC 网页支付）

#### 依赖
```xml
<dependency>
    <groupId>com.alipay.sdk</groupId>
    <artifactId>alipay-sdk-java-all</artifactId>
    <version>4.38.213.ALL</version>
</dependency>
```

#### 配置项
```yaml
payment:
  alipay:
    enabled: true
    app-id: "2021001234567890"
    gateway: https://openapi.alipay.com/gateway.do  # 生产；沙箱用 openapi.alipaydev.com
    private-key: your-app-rsa2048-private-key        # 应用私钥（PKCS8 格式）
    alipay-public-key: alipay-rsa2048-public-key    # 支付宝公钥
    return-url: https://yourdomain.com/payment/return
    notify-url: https://yourdomain.com/api/payment/notify/alipay
    charset: UTF-8
    sign-type: RSA2
```

#### 支付流程
```
1. 使用 AlipayTradePagePayRequest 构建请求：
   bizContent：{
     "out_trade_no": "{paymentNo}",
     "product_code": "FAST_INSTANT_TRADE_PAY",
     "total_amount": "299.00",
     "subject": "会议报名-{ticketName}"
   }
2. alipayClient.pageExecute(request, "GET") 返回完整 GET URL（非 HTML 表单）
   返回的 URL 存入 payment_records.pay_url，通过 PaymentResponse.payUrl 返回前端
3. 前端通过 window.open(payUrl, '_blank') 在新标签页跳转到支付宝收银台
   无需 document.write()，不破坏 SPA 路由

回调验签：
  - 从 request.getParameterMap() 提取所有参数
  - AlipaySignature.rsaCheckV1(params, alipayPublicKey, charset, signType)
  - 验签通过后：提取 out_trade_no、trade_no（支付宝流水号）、trade_status
  - trade_status=="TRADE_SUCCESS" 或 "TRADE_FINISHED" 时更新支付状态
  - 返回：HTTP 200 + 纯文本 "success"
```

#### 退款
```
使用 AlipayTradeRefundRequest：
bizContent：{
  "out_trade_no": "{paymentNo}",
  "refund_amount": "299.00",
  "out_request_no": "{refundNo}",
  "refund_reason": "管理员审批退款"
}
```

### 5.3 银联（UnionPay 在线支付）

#### 依赖
银联无官方 Maven 包，使用 [银联开放平台 SDK](https://open.unionpay.com/tjweb/support/apiList?apiServiceId=448)，手动引入 jar 或使用 HTTPClient 直接调用。

```xml
<!-- 使用 OkHttp + 手动实现签名 -->
<dependency>
    <groupId>com.squareup.okhttp3</groupId>

    <version>4.12.0</version>

```

#### 配置项
```yaml
payment:
  unionpay:
    enabled: true
    mch-id: "777290058110048"       # 商户号
    gateway: https://gateway.95516.com/gateway/api  # 生产
    cert-path: classpath:cert/unionpay_acp.pfx      # 商户证书（PFX 格式）
    cert-password: your-cert-password
    unionpay-cert-path: classpath:cert/unionpay_root.cer  # 银联根证书
    notify-url: https://yourdomain.com/api/payment/notify/unionpay
    return-url: https://yourdomain.com/payment/return
    front-url: https://yourdomain.com/payment/return  # 前台通知
```

#### 支付流程
```
1. 构建消费请求报文（PC 网页支付，txnType=01，channelType=07）
   参数：{
     "version": "5.1.0",
     "encoding": "UTF-8",
     "txnType": "01",         // 消费
     "txnSubType": "01",      // 自助消费
     "bizType": "000201",     // 互联网
     "channelType": "07",     // PC
     "backUrl": "{notifyUrl}",
     "orderId": "{paymentNo}",
     "txnAmt": "{金额（分）}",
     "currencyCode": "156"    // CNY
   }
2. 用商户 RSA 私钥签名整个报文
3. POST 到 https://gateway.95516.com/gateway/api/frontTransReq.do
4. 响应包含 HTML 表单，前端提交后跳转到银联收银台

回调验签：
  - 从 request 参数中提取 signature、certId 等
  - 用银联提供的公钥证书验 RSA 签名
  - 验证 respCode=="00" 表示支付成功
  - 返回：HTTP 200 + 空响应体
```

### 5.4 PayPal（Checkout Orders v2）

#### 依赖
```xml
<dependency>
    <groupId>com.paypal.sdk</groupId>
    <artifactId>paypalserversdk</artifactId>
    <version>0.6.1</version>
</dependency>
```

#### 配置项
```yaml
payment:
  paypal:
    enabled: true
    client-id: your-paypal-client-id
    client-secret: your-paypal-client-secret
    base-url: https://api-m.paypal.com    # 生产；沙箱用 api-m.sandbox.paypal.com
    webhook-id: your-webhook-id           # PayPal Dashboard 注册的 Webhook ID
    return-url: https://yourdomain.com/payment/paypal/return
    cancel-url: https://yourdomain.com/payment/paypal/cancel
    exchange-rate: 7.20                   # CNY→USD 固定汇率（建议定期更新配置）
```

#### 支付流程
```
1. 获取 Access Token（OAuth2 Client Credentials）：
   POST https://api-m.paypal.com/v1/oauth2/token
   Authorization: Basic {Base64(clientId:secret)}
   Body: grant_type=client_credentials

2. 创建 PayPal 订单：
   POST https://api-m.paypal.com/v2/checkout/orders
   {
     "intent": "CAPTURE",
     "purchase_units": [{
       "reference_id": "{paymentNo}",
       "description": "Conference Registration - {ticketName}",
       "amount": {
         "currency_code": "USD",
         "value": "41.53"         // CNY 金额 / 汇率
       }
     }],
     "payment_source": {
       "paypal": {
         "experience_context": {
           "return_url": "{returnUrl}?paymentNo={paymentNo}",
           "cancel_url": "{cancelUrl}"
         }
       }
     }
   }

3. 响应中提取 approve link：
   links[].rel == "payer-action" 的 href
   存入 payment_records.pay_url，返回前端

4. 前端跳转到 approve link，用户在 PayPal 页面授权

5. 用户授权后 PayPal 跳回 return_url，携带 token（PayPal Order ID）

6. 前端调用后端完成 Capture：
   POST /api/payment/paypal/capture?token={paypalOrderId}
   后端调用：POST https://api-m.paypal.com/v2/checkout/orders/{paypalOrderId}/capture

Webhook 事件验签：
  - PayPal 发送 POST 到 /api/payment/notify/paypal
  - 从 Header 提取：PAYPAL-TRANSMISSION-ID、PAYPAL-CERT-URL、PAYPAL-TRANSMISSION-SIG 等
  - 调用 PayPal Webhook 签名验证 API 或本地 RSA 验证
  - 监听事件：PAYMENT.CAPTURE.COMPLETED → 更新状态为 SUCCESS
```

> **注意**：PayPal 需要额外的 Capture 步骤（区别于国内的纯回调），建议同时支持：
> - 用户跳回时前端触发 `/api/payment/paypal/capture`（实时）
> - Webhook 的 PAYMENT.CAPTURE.COMPLETED 作为兜底保障

## 七、安全设计

### 7.1 回调端点安全

| 措施 | 说明 |
|------|------|
| 绕过 JWT 鉴权 | 在 `JwtAuthenticationFilter` 白名单中添加 `/api/payment/notify/*` |
| 强制验签 | 任何未通过签名校验的回调立即返回错误，不做任何业务处理 |
| 幂等处理 | 同一 `provider_trade_no` 的成功通知只处理一次（更新前检查状态） |
| IP 白名单（可选） | 微信/支付宝/银联均提供回调 IP 段文档，可在 Nginx 层限制 |
| HTTPS 强制 | 所有回调 URL 必须使用 HTTPS，各渠道拒绝 HTTP 回调地址 |

### 7.2 金额校验

```java
// 回调处理时必须校验金额，防止金额篡改攻击
BigDecimal notifyAmount = parseNotifyAmount(notifyData);
PaymentRecord record = paymentRecordMapper.selectByPaymentNo(paymentNo);
if (notifyAmount.compareTo(record.getAmount()) != 0) {
    log.error("金额不匹配！期望：{}，回调：{}", record.getAmount(), notifyAmount);
    return buildFailResponse();  // 拒绝该回调
}
```

### 7.3 重放攻击防护

微信 v3 回调中包含 `Wechatpay-Timestamp`，超过 5 分钟的回调拒绝处理。支付宝 SDK 内置时间戳校验。

## 十一、已知风险与注意事项

| 风险点 | 说明 |
|--------|------|
| 回调 URL 必须公网可达 | 开发阶段需使用 ngrok 等工具做内网穿透，不能用 localhost |
| 证书文件安全 | 微信私钥、支付宝私钥、银联 PFX 绝对不能提交 git，应通过环境变量或 Secret Manager 注入 |
| PayPal 汇率风险 | 固定汇率配置可能导致汇差损失，建议定期调整或对接汇率 API |
| 银联 SDK 无官方 Maven | 需要手动管理 SDK jar 文件，或使用 lib 目录引入 |
| 定时关单 | 超时 15 分钟的 PENDING 支付记录需要定时任务关闭（调用各渠道关单 API），需引入 `@Scheduled` |
| 退款到账异步 | 退款提交成功≠款项到账，`refund_records.status` 需通过渠道回调或查询接口异步更新 |
| 生产部署 | 各渠道的生产资质申请耗时较长（微信/支付宝约 1-3 个工作日，银联约 5-10 个工作日），需提前规划 |
