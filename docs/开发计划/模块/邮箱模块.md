## 一、背景与现状分析

### 1.1 当前邮件使用分布

系统现有 **6 处**邮件发送逻辑，全部硬编码在 Java 代码中：

| # | 场景 | 触发位置 | 接收方 | 当前状态 |
|---|------|---------|--------|---------|
| 1 | 邮箱验证码 | `AuthServiceImpl.register()` / `sendVerificationCode()` | 注册用户 | ✅ 已实现 |
| 2 | 投稿提交成功 | `SubmissionServiceImpl.create()` | 讲者 | ✅ 已实现 |
| 3 | 审稿分配通知 | `ReviewServiceImpl.assign()` | 审稿人 | ✅ 已实现 |
| 4 | 投稿审核结果 | `SubmissionServiceImpl.updateStatus()` | 讲者 | ✅ 已实现 |
| 5 | 订单创建成功 | — | 参会者 | ❌ 未实现 |
| 6 | 支付成功通知 | — | 参会者 | ❌ 未实现 |
| 7 | 凭证发放通知 | — | 参会者 | ❌ 未实现 |
| 8 | 会议提醒通知 | — | 参会者 | ❌ 未实现 |

### 1.2 现状问题

1. **内容不可配置**：所有邮件主题和 HTML 正文均硬编码于 `EmailServiceImpl.java`，修改模板须改代码、重新编译部署
2. **账号绑定混乱**：业务邮件（`EmailServiceImpl`）读取 `application.yml` 静态配置；测试邮件（`EmailConfigServiceImpl`）读取数据库 `email_config` 表。两套系统各自为政，维护成本高
3. **缺少场景覆盖**：订单、支付、凭证类邮件尚未实现，与需求文档不符
4. **无可视化管理**：管理员无法在后台预览、编辑、测试任意场景的邮件

### 1.3 需求目标

在管理端新增「邮件模板管理」页面，实现：

- **声明式配置**：每个邮件场景可独立指定发送账号（绑定某条 `email_config`）和自定义模板（主题 + HTML 正文）
- **变量占位符**：模板支持 `{{variableName}}` 占位符，运行时自动替换为业务数据
- **在线预览/测试**：管理员可直接在页面预览渲染后的效果，并发送测试邮件到指定地址
- **统一入口**：业务邮件和配置测试邮件统一通过模板系统发送，消除双轨并存问题
- **多邮箱配置支持**：系统支持维护多个邮箱账号，支持启用/禁用、设置默认账号，模板可按场景绑定指定账号
- **发送日志可追踪**：记录每次邮件发送的收件人、场景、账号、状态、错误信息与发送时间，支持后台查询

## 二、数据库设计

### 2.1 新增表：`email_template`

```sql
CREATE TABLE `email_template` (
  `id`             BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键',
  `scene_code`     VARCHAR(64)   NOT NULL COMMENT '场景码（全局唯一，英文大写下划线）',
  `scene_name`     VARCHAR(100)  NOT NULL COMMENT '场景中文名称',
  `email_config_id` BIGINT       DEFAULT NULL COMMENT '绑定的邮箱配置 ID（NULL 时使用系统默认启用配置）',
  `subject`        VARCHAR(200)  NOT NULL COMMENT '邮件主题（支持 {{变量}} 占位符）',
  `body`           MEDIUMTEXT    NOT NULL COMMENT '邮件正文 HTML（支持 {{变量}} 占位符）',
  `variables_desc` JSON          DEFAULT NULL COMMENT '可用变量说明，格式：[{"name":"speakerName","desc":"讲者姓名"}]',
  `is_enabled`     TINYINT       NOT NULL DEFAULT 1 COMMENT '是否启用：1=启用，0=禁用',
  `created_at`     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at`     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_scene_code` (`scene_code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='邮件模板表';
```

**实现要点**：`EmailTemplateServiceImpl.java`

```java
@Service
@RequiredArgsConstructor
public class EmailTemplateServiceImpl implements EmailTemplateService {

    private final EmailTemplateMapper emailTemplateMapper;
    private final EmailConfigMapper emailConfigMapper;
    private final AesUtil aesUtil;

    /** 占位符替换：将 {{key}} 替换为 variables.get(key) */
    private String render(String template, Map<String, String> variables) {
        if (template == null || variables == null) return template;
        String result = template;
        for (Map.Entry<String, String> entry : variables.entrySet()) {
            result = result.replace("{{" + entry.getKey() + "}}",
                                    entry.getValue() == null ? "" : entry.getValue());
        }
        return result;
    }

    @Override
    public void send(String sceneCode, String to, Map<String, String> variables) {
        EmailTemplate template = getBySceneCode(sceneCode);
        if (template == null || template.getIsEnabled() != 1) {
            log.warn("[EmailTemplate] 场景 {} 模板未配置或已禁用，跳过发送", sceneCode);
            return;
        }

        String subject = render(template.getSubject(), variables);
        String body    = render(template.getBody(),    variables);

        // 选择发送账号：模板绑定 > 系统默认启用账号
        EmailConfig config = template.getEmailConfigId() != null
            ? emailConfigMapper.selectById(template.getEmailConfigId())
            : emailConfigMapper.selectOne(
                new LambdaQueryWrapper<EmailConfig>()
                    .eq(EmailConfig::getIsEnabled, 1).last("LIMIT 1"));

        if (config == null) {
            log.error("[EmailTemplate] 场景 {} 无可用邮箱配置，发送失败", sceneCode);
            return;
        }

        // 构建 JavaMailSender 并发送（复用 EmailConfigServiceImpl 的 buildMailSender 逻辑）
        sendViaConfig(config, to, subject, body);
    }
}
```

### 3.6 EmailService 改造

将现有 `EmailServiceImpl` 中所有硬编码模板替换为调用 `EmailTemplateService.send()`：

| `sendReviewAssignment(to, reviewerName, title)` | `templateService.send("REVIEW_ASSIGNMENT", to, {...})` |

## 十一、v1.1 增量需求设计（多邮箱配置 + 邮件发送日志）

### 11.1 增量目标

- 邮箱配置从“单条系统配置”升级为“多账号管理”，支持新增、编辑、启停、设置默认账号
- 邮件模板可绑定任意可用邮箱账号；未绑定时走默认账号
- 每次邮件发送（业务发送 + 测试发送）都写入日志，记录是否成功与失败原因
- 后台支持按条件查询发送日志，用于审计和问题排查

### 11.2 数据库设计增量

#### 11.2.1 `email_config` 增量字段（支持多账号管理）

> 现有表已具备多行存储能力，本次补充“配置名称、默认标记、优先级”等管理字段。

```sql
ALTER TABLE `email_config`
  ADD COLUMN `config_name` VARCHAR(100) NOT NULL DEFAULT '' COMMENT '配置名称' AFTER `id`,
  ADD COLUMN `is_default` TINYINT NOT NULL DEFAULT 0 COMMENT '是否默认：1=默认，0=非默认' AFTER `is_enabled`,
  ADD COLUMN `priority` INT NOT NULL DEFAULT 100 COMMENT '优先级（越小越高）' AFTER `is_default`,
  ADD COLUMN `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注' AFTER `priority`;

CREATE INDEX `idx_email_config_enabled_default` ON `email_config` (`is_enabled`, `is_default`, `priority`);
```

约束建议：
- 通过服务层事务保证同一时刻仅一条 `is_default=1` 且 `is_enabled=1` 的记录
- 禁止删除唯一默认账号（或删除前先切换默认账号）

#### 11.2.2 新增表：`email_send_log`（邮件发送日志）

```sql
CREATE TABLE `email_send_log` (
  `id`              BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键',
  `scene_code`      VARCHAR(64)   NOT NULL COMMENT '场景码，如 VERIFICATION_CODE',
  `template_id`     BIGINT        DEFAULT NULL COMMENT '模板ID',
  `email_config_id` BIGINT        DEFAULT NULL COMMENT '发送使用的邮箱配置ID',
  `send_type`       TINYINT       NOT NULL COMMENT '发送类型：1=业务发送，2=测试发送',
  `to_email`        VARCHAR(200)  NOT NULL COMMENT '收件人邮箱',
  `from_email`      VARCHAR(200)  DEFAULT NULL COMMENT '发件人邮箱',
  `subject`         VARCHAR(300)  DEFAULT NULL COMMENT '发送主题（渲染后）',
  `variables_json`  JSON          DEFAULT NULL COMMENT '变量快照',
  `status`          TINYINT       NOT NULL COMMENT '发送状态：1=成功，0=失败',
  `error_message`   VARCHAR(1000) DEFAULT NULL COMMENT '失败原因',
  `triggered_by`    BIGINT        DEFAULT NULL COMMENT '触发人（管理员ID，业务触发时可空）',
  `sent_at`         DATETIME      DEFAULT NULL COMMENT '发送时间',
  `created_at`      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_scene_code` (`scene_code`),
  KEY `idx_to_email` (`to_email`),
  KEY `idx_status` (`status`),
  KEY `idx_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='邮件发送日志表';
```

### 11.3 后端实现增量

#### 11.3.1 多邮箱配置管理

新增/改造服务：
- `EmailConfigService`：支持列表查询、新增、编辑、启停、设为默认、删除
- `EmailTemplateService`：编辑模板时读取邮箱配置列表，不再依赖单条系统配置

账号选择策略（发送时）：
1. 模板绑定了 `email_config_id`：优先使用该账号（不存在或禁用则报错）
2. 模板未绑定账号：使用 `is_default=1 AND is_enabled=1` 的默认账号
3. 默认账号不存在时：按 `is_enabled=1` 且 `priority ASC` 兜底一条
4. 仍无可用账号：返回 `EMAIL_TEMPLATE_NO_CONFIG(8012)`
