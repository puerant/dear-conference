## 二、核心业务流程

### 2.1 讲者投稿

```
前端 Portal /submission/create
  → POST /api/submissions { title, abstract, category, fileUrl? }
  → @RequireRole("speaker") 校验
  → SubmissionServiceImpl.create()
      - INSERT INTO submissions (user_id, title, abstract, category, file_url, status=PENDING)
  → 返回新建 Submission 对象

关键字段说明：
  - abstract 字段：Java 关键字冲突，实体中命名为 abstract0，
    通过 @TableField("`abstract`") + @JsonProperty("abstract") 映射
  - status 初始值：PENDING（DB 存 1，JSON 序列化为字符串 "pending"）
```

---

### 2.2 管理员分配审稿人

```
前端 Admin 投稿列表 → 点击「分配」
  → 弹框调用 GET /api/admin/users?role=reviewer 加载审稿人列表
      - 若无审稿人 → 显示「暂无审稿人」空状态，禁用确认按钮
  → 管理员勾选审稿人后确认
  → PUT /api/admin/submissions/{id}/assign { reviewerIds: [1,2,...] }
  → AdminSubmissionController.assignReviewer()
      - 将投稿 status 更新为 REVIEWING（DB 存 2，JSON 序列化为 "reviewing"）
  → 返回 200
```

**状态流转：**
```
pending ──[分配审稿人]──> reviewing ──[管理员操作]──> accepted
                                                   └──> rejected
```

---

### 2.3 审稿人提交审稿

```
前端 Portal /review
  → GET /api/reviews/my 获取当前审稿人已提交的审稿记录列表
  → ⚠️ 注意：审稿人无法从该列表发现新分配的待审稿投稿（Bug-14，assign 不创建 review 记录）
  → 直接访问 /review/:id（需提前知道投稿 id）
      - 并行请求：GET /api/submissions/{id}（论文信息）
                  GET /api/reviews/submission/{id}（已有审稿记录）
      - 若已提交过审稿 → 展示审稿结果（只读）
      - 若未审稿 → 显示审稿表单
  → POST /api/reviews { submissionId, result, score, comment }
      - result 值：1=accept / 2=reject / 3=revise（提交时为数字）
      - 后端 ReviewResult 枚举 @JsonValue 序列化为字符串 code
      - ⚠️ 副作用：ReviewServiceImpl.create() 会根据 result 自动更新 submission.status
          → result=ACCEPT → submission.status = ACCEPTED（3）
          → result=REJECT → submission.status = REJECTED（4）
  → 返回 200
```

---

### 2.4 管理员更新投稿最终状态

```
前端 Admin 投稿详情页
  → PUT /api/admin/submissions/{id}/status { status: "accepted" | "rejected" }
  → AdminSubmissionController.updateStatus()
      - SubmissionStatus.ofCode(statusCode) 将字符串转为枚举（已修复）
      - 更新 submissions.status
  → 讲者端刷新投稿详情，状态标签随之变化
```

---

### 2.5 枚举序列化规则（重要）

所有状态枚举（`SubmissionStatus`、`ReviewResult`）均采用统一模式：

| 层面 | 存储 / 传输值 | 示例 |
|------|------------|------|
| MySQL 数据库 | `@EnumValue` 标注的 `Integer value` | `1`, `2`, `3` |
| JSON 接口响应 | `@JsonValue` 标注的 `String code` | `"pending"`, `"accept"` |
| 前端 TypeScript | `string` 类型 | `"pending"`, `"accept"` |
| 前端 format.ts | 字符串 key 映射 | `{ pending: '待审稿' }` |
